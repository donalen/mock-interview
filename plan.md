# План: веб-игра «Пинг-понг» на двоих (React + TypeScript)

## 1. Технологии и обоснование

| Слой | Выбор | Почему |
|------|-------|--------|
| UI-фреймворк | **React 19 + TypeScript** | требование ТЗ; типобезопасные сущности `Game/Board/Paddle/Ball/Player` |
| Сборка/окружение | **существующий `StarterKit` (Astro 5 + @astrojs/react)** | в репозитории уже настроены React + TS; не изобретаем сборку. Игра — React-остров (`client:only="react"`) |
| Графика | **Canvas 2D API** | см. ниже |
| Игровой цикл | **`requestAnimationFrame`** | стабильные ~60 fps, синхронизация с частотой монитора |
| Состояние кадра | **`useRef`** (не `useState`) | 60 ре-рендеров React в секунду — антипаттерн; refs держат физику вне рендера |

### Почему Canvas 2D, а не DOM/SVG/WebGL
- **DOM (div+CSS)** и **SVG**: анимация 60 fps через React-стейт = перерисовка дерева каждый кадр → лаги; физика в пикселях через layout неудобна.
- **WebGL / Three.js / PixiJS**: избыточно — на поле всего 2 прямоугольника, круг и линия.
- **Canvas 2D**: один императивный цикл отрисовки вне React, точный контроль кадра, минимум примитивов. Оптимальный баланс простоты и производительности.

### Принцип разделения
- React отвечает за **экраны и «редкое» состояние**: текущий экран (старт/игра/финал), счёт, таймер, имена.
- `<canvas>` + `requestAnimationFrame` отвечают за **игровое поле**: координаты и движение мяча/паддлов.
- React-стейт обновляется только на событиях: гол, тик таймера, конец матча.

## 3. Модель данных (типы по ТЗ)

```ts
type Player = { id: 1 | 2; name: string };
type Ball   = { x: number; y: number; vx: number; vy: number; speed: number; radius: number; color: string };
type Paddle = { x: number; y: number; width: number; height: number; speed: number };
type Board  = { width: number; height: number; color: string; wallThickness: number };
type Score  = { p1: number; p2: number };
type Game   = { board: Board; players: [Player, Player]; score: Score; timeLeft: number };
```

## 4. Игровая логика (ключевые правила)

- **Старт мяча**: направление задаётся случайно (знак vx/vy).
- **Стенки**: рикошет от верхней и нижней (инверсия vy).
- **Паддлы**: игрок 1 — `W`/`S`, игрок 2 — `↑`/`↓`; движение по вертикали с `clamp` в пределах поля.
- **Отбивание**: при столкновении с паддлом vx инвертируется; угол отскока зависит от точки касания (аркадная физика классического Pong).
- **Гол**: мяч ушёл за левый/правый край → очко сопернику пропустившего.
- **Подача**: мяч ставится на паддл пропустившего игрока, ждёт **3 секунды**, затем летит в сторону соперника.
- **Сетка**: вертикальная линия по центру (декоративная).
- **Конец**: истёк таймер матча → `WinnerModal` с победителем (или «Ничья» при равенстве).

## 5. Этапы реализации

1. **Каркас**: страница-остров, типы, палитра, `StartScreen` (время/имена/Старт), переключение экранов `start → play → end`.
2. **Поле**: `GameCanvas` во весь экран (чуть меньше окна), зелёный фон, верх/низ стенки, центральная сетка, статичные мяч и паддлы.
3. **Петля**: `useAnimationFrame`, полёт мяча в случайном направлении, рикошет от стенок.
4. **Управление**: `useKeyboard` (WASD + стрелки), движение паддлов с ограничением по краям.
5. **Отбивание**: коллизии мяча с паддлами, угол отскока от точки удара.
6. **Голы и подача**: детект вылета за край, начисление очка, постановка мяча на паддл и подача через 3 сек.
7. **HUD**: `ScoreBoard` (таймер + счёт справа вверху), `PlayerLabels` (имена внизу).
8. **Таймер и финал**: обратный отсчёт матча, `WinnerModal` по завершении.
9. **Полировка**: адаптив к ресайзу окна, стили, кнопка рестарта.

## 6. Согласованные решения
- Игра встраивается в существующий `StarterKit` (Astro + React); запуск через `npm run dev`, страница `/pong`.
- «Время игры» на старте задаётся **в минутах**, внутри переводится в секунды.
- Матч идёт **только до конца таймера** — лимита очков для досрочной победы нет; победитель — у кого больше очков.
- При равном счёте по окончании — «Ничья».
- Угол отскока управляется точкой касания паддла (классический аркадный Pong).
